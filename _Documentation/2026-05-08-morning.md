## Open questions

1. **Which is the "site"?** The faithful port, the map version, both with
   different framing, or one as the entry and the other as a reference?
   - a. Faithful port is canon, map is filed under archive/. Strongest claim
     to preservation; the map version becomes a footnote.
   - b. Map version is canon, faithful port is the archive. Easier to read,
     but loses the original's distinct look.
   - c. Both at the root, with a small index page that says "two readings."
     Most honest about what the revival is, costs one extra page.
   - d. Faithful port at root, map version promoted to `archive/map.html`
     with a clearer link from the footer. Closest to current state, lowest
     friction.

2. **Preserved bugs as features.** The lat/lon label-swap and the squashed
   non-Mercator projection are kept on purpose right now. Is that the rule
   for everything, or is there a list of "preserve" vs "fix"? Worth writing
   down before another session relitigates it.

3. **What does `archive/` contain?** `mov_frames_2fps/` (reference, can
   probably go), the SWF, the disassembly, the original 2004 HTML, the
   no-images variant, the map version. Mixed: some are source, some are
   debris. A short README inside `archive/` would stop the next session
   guessing.

4. **EXIF time assumption.** The code comments "Iceland keeps GMT year-round
   so EXIF == UTC." Iceland uses GMT, true; but the camera was Timo's, set
   to camera-local time which could have been CEST. Worth a one-photo
   sanity check (pick a photo with a known sun position, see if the GPS
   trkpt at that EXIF time matches where the photo looks like it was
   taken).

5. **No CLAUDE.md.** The port's hand-won knowledge (spring constants,
   twip-to-px conversion, the 1.2/frame fade rate, the SWF's fps×rate=time
   relationship) lives in code comments only. A short `CLAUDE.md` at the
   root pointing at the disassembly and naming the "preserve vs fix" list
   would survive better than a comment block.
